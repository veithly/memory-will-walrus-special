# Product

## Register

product

## Users

Memory Will serves teams running long AI-assisted work where context loss or unsafe handoff has real cost. The beachhead user is a research or product lead who needs a successor agent to continue a vendor comparison after the first agent is intentionally killed.

## Product Purpose

The product turns an agent handoff into a controlled inheritance event. A user kills a predecessor agent, scopes what can survive, writes a durable Memory Will, opens a cold successor, and inspects a receipt proving what memory was restored and what was withheld.

## Brand Personality

Tense, exact, forensic. The interface should feel like a succession control surface for serious work, not a chat app, proof dashboard, or storage demo.

## Anti-references

Do not resemble a generic AI chat shell, blockchain explorer, admin analytics dashboard, shadcn card wall, proof packet, or pitch-deck landing page. The first screen must be the workbench and kill/restore state change, not marketing copy.

## Design Principles

- Death before documentation: the first action is the state change.
- Scope is visible: allowed and unavailable memory are shown side by side.
- Proof stays inspectable but secondary: receipt and pointer are present without swallowing the hero path.
- One destructive action, one recovery result: every screen supports the kill, restore, mutate, inspect loop.
- Honest failure earns trust: storage, provider, stale checkpoint, and restore failures block success.

## Accessibility & Inclusion

Target WCAG AA contrast, keyboard-operable controls, 44px minimum tap targets, clear destructive-action confirmation, and a `prefers-reduced-motion` fallback for the kill/restore animation. Color cannot be the only indicator for alive, dead, writing, restored, blocked, or failed state.
