# Concept Lock

## Selected Scene

- Product working name: Memory Will
- Source candidate: S1, derived from I3-2 Agent Funeral / Memory Will.
- Judge consensus: J1 advanced S1 as strongest Walrus sponsor fit, J2 kept S1 as a top product backup with a retellable human hook, and J3 tied S1 for strongest build/proof score.
- Seven-word rumor: Kill one agent; another inherits verified memory.

## Full Concept Lock

- User: teams running long AI work that cannot die with one chat, contractor, browser tab, or teammate. Beachhead user is a research/product lead who needs a successor agent to finish a vendor comparison after the first agent is killed.
- Required construct primer: Sui Overflow `Special - Walrus` asks for functional AI agents or agent workflows using Walrus as verifiable durable data infrastructure. This concept counts only if a judge-triggered action writes a durable Memory Will to Walrus/MemWal, then a cold successor agent reopens scoped memory, resumes work, and shows inspectable proof. It does not count if the successor sees hidden local context, pasted transcript, static summary, or frontend-only state.
- Current pull: AI agents are moving from novelty chats to long-running work, but everyone has felt context loss, tab death, tool lock-in, and untrusted handoffs. People now want durable agent state they can inspect, scope, delegate, restore, and continue across tools.
- Desire / power pull: "My work survives the agent doing it." The user controls what memory survives, what is redacted, and what a successor is allowed to inherit.
- Cool utility: press `Kill Agent`, choose a successor role, and watch a blank agent resume from a scoped durable memory object instead of a messy chat dump.
- Tuesday pull: after any research sprint, sales call, client project, diligence task, or long agent run, the user can create or update a Memory Will so the work can survive app resets and teammate handoffs.
- Public flex artifact: a Memory Will Receipt showing killed-agent state, delegated memories, redactions, successor replay, Walrus/MemWal pointer, and before/after output.
- Product thesis: long-running agent work needs a portable succession layer where work-in-progress memory, evidence, policies, and checkpoints can outlive the original agent and be inherited by another role or tool.
- Beachhead demo case: a predecessor research agent is halfway through comparing three vendors. The judge kills it, selects a successor role such as finance, legal, designer, or operator, and the successor resumes from the Walrus-backed Memory Will with role-specific focus and visible redactions.
- Three adjacent cases same engine can handle: sales account handoff, legal matter handoff, research sprint continuation.
- Reusable agent/product engine: memory will writer, scoped memory/delegation policy, durable checkpoint store, cold successor restore, redaction checker, replay receipt renderer.
- Expansion ladder: beachhead is one killed research agent; month 1 is team handoff spaces with scoped memory wills; month 6 is a cross-tool memory succession layer for enterprise agent workflows.
- Why a non-domain judge personally cares: every judge who has used AI tools understands losing context or handing messy work to another person. Killing the agent makes the pain and recovery obvious without a storage lecture.
- Familiar desire/pressure hook in first 5 seconds: the app invites the judge to kill an in-progress AI task and see whether the work survives.
- Judge roleplay action: press `Kill Agent`, choose a successor role, optionally block one memory category, then open the successor.
- Non-domain retell line: "I killed an AI, it wrote a memory will, and the next AI continued only with the memories I allowed."
- Public demo hook: the room can choose which role inherits the work and which memory category gets redacted.
- Industry buyer behind it: AI workflow platforms, consulting firms, sales organizations, research teams, law firms, and internal enterprise AI automation teams.
- One visible action: kill the predecessor agent and choose the successor scope.
- Judge first action <=30s: click `Kill Agent`, then choose finance, legal, designer, or operator as successor.
- Agent-native loop if agent contest: predecessor agent pursues a research comparison goal, observes source facts and task progress, writes durable checkpoint/memory, dies on judge command, successor observes scoped Memory Will, plans next step, recalls allowed state, verifies sources and redactions, continues or asks for missing memory.
- Agent tools >=3: Walrus/MemWal write/read/restore/delegate; source/artifact opener; comparison compiler; redaction policy checker; verification/replay renderer.
- Durable state objects >=3: Memory Will object, task checkpoint, evidence/source bundle, delegation policy, redaction log, successor replay trace, final comparison artifact.
- Surprise/replan path: judge blocks pricing notes or switches successor from finance to legal. The successor must acknowledge the missing scope, use only allowed memories, and produce a narrower output.
- Before: a predecessor agent has volatile context and in-progress work that would normally vanish or be pasted as a messy transcript.
- After: a cold successor agent restores from a scoped Walrus/MemWal object and produces role-specific continuation with inspectable proof of what it inherited.
- Result artifact: Memory Will Receipt plus resumed comparison artifact.
- 5-second replayable clip: agent flatlines, Memory Will stamp appears, local context clears, successor opens the will, output resumes with the chosen role's bias.
- What changes when judge changes input: successor role changes delegated memory and output priorities; blocked memory category changes redactions and uncertainty; kill timing changes checkpoint and recovery trace.
- Why now: agentic work is becoming longer-lived, multi-tool, and team-facing; Walrus/MemWal makes durable portable memory, restore, and delegation a sponsor-native primitive instead of hidden app state.
- Why sponsor/domain primitive is necessary: without durable Walrus/MemWal memory, the successor either inherits hidden local context or receives an untrusted summary. The product consequence is scoped restore, not storage.
- What judges can inspect: Memory Will JSON/object, Walrus/MemWal pointer, delegated memory scopes, redaction entries, checkpoint before death, successor replay trace, and restored output.
- Boring clone avoided: project handoff summary, chat export, knowledge-base article, workflow log viewer.
- Mutation that makes it not boring: make the judge kill the agent, choose who inherits, and watch a cold successor continue from scoped durable memory.
- Three cuts: no general agent framework, no real enterprise identity integration, no broad vertical workflows beyond the research-comparison beachhead.
